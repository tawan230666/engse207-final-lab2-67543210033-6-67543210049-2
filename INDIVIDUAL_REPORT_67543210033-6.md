ส่วนที่รับผิดชอบ
พัฒนา Auth Service และเพิ่มระบบลงทะเบียน (Register API)

ปรับแต่ง Log System ใน Auth Service ให้บันทึกลงตาราง logs ใน Database ของตัวเอง

จัดทำไฟล์ init.sql เพื่อ Seed บัญชีผู้ดูแลระบบ (Admin) ให้พร้อมใช้งานตั้งแต่รันระบบครั้งแรก

Deploy Auth Service และ Auth-DB บน Railway

สิ่งที่ลงมือทำจริง
ทำการเขียนโค้ดเพิ่มเติมใน routes/auth.js เพื่อรับข้อมูลการสมัครสมาชิก ตรวจสอบความซ้ำซ้อนของอีเมล/ชื่อผู้ใช้ และทำการเข้ารหัสรหัสผ่านด้วย bcryptjs ก่อนบันทึกลงฐานข้อมูล นอกจากนี้ได้รื้อถอน Log Service ของ Set 1 ทิ้ง และเขียนฟังก์ชัน logEvent ใน Auth Service ใหม่ เพื่อดักจับ Event ต่างๆ (เช่น LOGIN_SUCCESS, REGISTER) และ Insert ข้อมูลลง Database ตัวเอง พร้อมทั้งสร้าง API /api/auth/logs สำหรับให้ Admin ดึง Log ไปแสดงผลบนหน้าจอ

ปัญหาที่พบและวิธีแก้
ปัญหา: Nginx Container ในเครื่อง Local เกิดการ Crash วนลูป (Exited 1) เมื่อพยายามสร้าง SSL Certificate
วิธีแก้: ยกเลิกการใช้งาน Nginx และ HTTPS ในเครื่อง Local (ใช้การรันเปิดพอร์ตตรง 8080 แทน) เพื่อให้โฟกัสกับการพัฒนา Backend ได้เต็มที่ ส่วนการทำ HTTPS ได้นำไปให้ Railway Cloud จัดการให้โดยอัตโนมัติในระดับ Production

ปัญหา: ข้อมูล Log ไม่แสดงผลบนหน้าเว็บ หลังจากรื้อ Log Service ออก
วิธีแก้: เขียน API GET /api/auth/logs เพิ่มเติมในตัว Auth Service เอง และปรับแต่งหน้าเว็บ logs.html ให้ยิง API ไปที่ Auth Service แทน รวมถึงมีการตรวจสอบ Role ว่าต้องเป็น Admin เท่านั้นจึงจะเรียกดูได้

สิ่งที่เรียนรู้เชิงสถาปัตยกรรมจาก Set 2
ได้เรียนรู้หลักการทำงานแบบ Database-per-Service อย่างลึกซึ้ง ซึ่งช่วยลดปัญหาคอขวด (Bottleneck) ที่เกิดจากการใช้ Database ตัวเดียวร่วมกันหลาย Service รวมถึงได้เห็นความสำคัญของการใช้ JWT ในการแชร์ Context ระหว่าง Microservices โดยไม่ต้องไปแตะฐานข้อมูลของ Service อื่น

ส่วนที่ยังไม่สมบูรณ์หรืออยากปรับปรุง
การจัดการข้อมูล Log เมื่อปริมาณเพิ่มมากขึ้นเรื่อยๆ อาจทำให้ตารางบวมได้ ในอนาคตควรมีการทำ Log Rotation หรือระบบลบ Log เก่าทิ้งอัตโนมัติเมื่อครบ 30 วัน
